# MacroPad updater release guide

MacroPad Studio contains two independent signed update paths:

- Tauri's signed desktop updater for Windows NSIS/MSI, macOS app archives, and Linux AppImage releases.
- A signed ESP32-S3 firmware package updater using the embedded Rust `espflash` library. Users do not install Python, PlatformIO, or a sidecar flasher.

## One-time release-key setup

Generate the updater key outside the repository and back it up securely:

```powershell
npm.cmd run tauri -- signer generate -w "$env:USERPROFILE\.tauri\macropad.key"
```

Never commit `macropad.key`, its password, or `TAURI_SIGNING_PRIVATE_KEY`. The generated public key is safe to embed. The private key is required for every future desktop and firmware update; losing it prevents updates to existing installations.

## Distribution-build configuration

Set these variables before compiling a distributable application:

```powershell
$env:MACROPAD_UPDATE_PUBLIC_KEY='<base64 public key generated by Tauri>'
$env:MACROPAD_APP_UPDATE_ENDPOINT='https://github.com/OWNER/REPOSITORY/releases/latest/download/latest.json'
$env:MACROPAD_FIRMWARE_MANIFEST_URL='https://github.com/OWNER/REPOSITORY/releases/latest/download/firmware-latest.json'
$env:TAURI_SIGNING_PRIVATE_KEY_PATH="$env:USERPROFILE\.tauri\macropad.key"
$env:TAURI_SIGNING_PRIVATE_KEY_PASSWORD='<key password>'
npm.cmd run tauri -- build
```

The endpoint and public key are compiled into the Rust binary. Development builds without them remain functional but show that release channels are not configured. Both endpoints must use HTTPS.

## GitHub-only automatic releases

No separate server is required. The repository includes
`.github/workflows/release.yml`, which builds Windows, Linux, both macOS
architectures, and the USB firmware before publishing one combined GitHub
Release.

Configure these values once under **Repository settings → Secrets and
variables → Actions**:

| Type | Name | Value |
|---|---|---|
| Variable | `MACROPAD_UPDATE_PUBLIC_KEY` | The public key printed by the Tauri signer |
| Secret | `TAURI_SIGNING_PRIVATE_KEY` | The complete private-key contents |
| Secret | `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | The private-key password |

The workflow automatically uses these public endpoints:

```text
https://github.com/Lumslucky/MacropadSoftware/releases/latest/download/latest.json
https://github.com/Lumslucky/MacropadSoftware/releases/latest/download/firmware-latest.json
```

The repository must be public for installations to download releases without
credentials. The private signing key never belongs in the repository.

To publish, synchronize the application version in `package.json`,
`src-tauri/Cargo.toml`, and `src-tauri/tauri.conf.json`, commit and push the
changes, then push the matching tag:

```powershell
git tag v3.0.0
git push origin v3.0.0
```

The release stays as a draft until every desktop target and the firmware have
built and been signed. The final job attaches the firmware manifest and package,
publishes the release, and marks it as latest. A failed build is never promoted.

## Publish a desktop release

1. Set the same semantic version in `package.json`, `src-tauri/tauri.conf.json`, and `src-tauri/Cargo.toml`.
2. Run all frontend and Rust tests.
3. Build on Windows, macOS, and Linux with the signing environment above.
4. Upload the generated updater bundles and `.sig` files without renaming or modifying them.
5. Publish `latest.json` using Tauri's static updater format. `tauri-action` can generate this file from the platform matrix.
6. Test an installed previous signed version before promoting the release.

Tauri rejects unsigned or incorrectly signed application updates. Windows uses passive installer mode; the app exits during installation and restarts on completion.

## Publish a firmware release

First update the firmware version constants in `firmware/src/main.cpp`, then run:

```powershell
.\scripts\Build-FirmwareRelease.ps1 `
  -Version 0.2.0 `
  -PackageUrl 'https://github.com/OWNER/REPOSITORY/releases/download/firmware-v0.2.0/macropad-firmware-0.2.0.zip' `
  -Notes 'OLED stability, lighting mapping, and updater support.'
```

Upload both generated files from `release/firmware/<version>/`:

- `macropad-firmware-<version>.zip`
- `firmware-latest.json`

The `.sig` file may also be retained as a release audit artifact. The ZIP contains the authoritative PlatformIO flash layout:

| Image | Offset |
|---|---:|
| `bootloader.bin` | `0x0000` |
| `partitions.bin` | `0x8000` |
| `boot_app0.bin` | `0xE000` |
| `firmware.bin` | `0x10000` |

The signed ZIP also contains `version.txt`; Device Center requires it to match the manifest, preventing an older authentic firmware package from being relabeled as a newer release.

Before any write, the desktop verifies HTTPS transport, package size, SHA-256, the pinned publisher signature, required ZIP entries, ESP32-S3 image headers, USB VID/PID, and ROM-reported chip type. `espflash` verifies the written segments and hard-resets the board.

## Initial firmware transition

Firmware versions before `0.2.0` do not answer the HID version query. Flash the first updater-aware build once using PlatformIO or the verified Device Center flow. From `0.2.0` onward, Device Center confirms the running version after reboot.

## Recovery

If flashing is interrupted, the ESP32 ROM bootloader remains recoverable:

1. Hold BOOT0.
2. Press RESET or reconnect USB.
3. Release BOOT0 after the serial port appears.
4. Reopen Device Center, download/verify the release again if necessary, select the detected port, and retry.

Never mutate an already published artifact or reuse a version. Roll forward with a higher signed version.
